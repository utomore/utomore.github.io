DataStructure 1 - Intro

Data Structure 系列

This is Series of Data Structure.

因為最近非本科轉換領域的關係，以此來記錄學習的歷程，主要學習的資料來源為 Udemy 上的課程。

  • 名稱: Mastering Data Structures & Algorithm using C/C++
  • 老師: Adbul Bari
  • 時長: 58 hr 30 min

0. Introduction 基本介紹

資料結構表達出如何有組織地將資料放置於記憶體中。主要分類為:

  • 實際資料結構(Physical Data Structure)
    • 陣列 Array
    • 串列 Linked List
  • 邏輯資料結構(Logical Data Structure)
    • 堆疊 Stack
    • 駐列 Queue
    • 樹 Trees
    • 圖 Graph
    • 哈希表 Hashing
    • 遞迴 Recursion

好的資料結構規劃，搭配上演算法可以讓資料存取上更加的有效率。最後會簡單說明上方這些資料結構的基本特性。

學習資料結構的等級

  1. 知道資料結構如何運作
  2. 知道執行時的消耗、可以有效的分析該資料結構的效率
  3. 可以自行創建有利於情況的資料結構

這系列學習筆記使用 C/C++ 來實作資料結構

1. 介紹資料儲存

在世界上的資料，目前有四種主要的處理方式，其中對應到不同的技術與演算法。分別為:

資料結構 (Data structure)

  • 如何有效率的儲存/組織主記憶體上的資料
  • 如何有效率的存取主記憶體中的資料

資料庫 (Storage)

  • 如何將資料放置於硬碟 (Hard Disk) 中，並採用標準的格式，使主記憶體更容易從第三類儲存設備中存取資料
  • 重點在於如何設計資料庫結構

資料倉儲 (Data Warehouse)

  • 儲存大量不常使用的資料 (Legacy data or historical data)
  • 由大量的硬碟陣列 (Array of Disk)磁片儲存
  • 目的儲存10年、50年以上
  • 專門處理此種資料的演算法: 資料探勘 (Data Mining Algorithm)

大數據(Big Data)

  • 藉由網路為主要的媒介，蒐集各式各樣的資料並處理
  • 主要演算法: 機器學習

一般程式執行時，處理的流程為: 硬碟(Hard Disk) > 主記憶體 > CPU 暫存器 > CPU 處理。

2. 主記憶體 Stack & Heap

記憶體最小單位 Bytes，每一個 Bytes 都有自己的位址編號(線性唯一編號)。現今的記憶體會進行主動切割為 Segement 當作一個可管理的記憶體單元(64kB = 64 * 1024 = 65536)

程式碼如何使用記憶體

一個 Segement (0 ~ 65535)，主要被分為三個部分。其中程式碼中變數所佔的 Bytes 數是根據編譯器與作業系統來決定。

  • Heap
  • Stack
  • Code Section

Stack

  1. 使用範圍: 記憶體編號低到高
  2. 函數在**編譯時期(Compiler time)**決定需要多少 Bytes 的記憶體空間
  3. 當函數呼叫時，自動分配；當函數結束時，自動釋放
  4. 程式可直接存取 (stack 與 code section 資源)

Heap

名稱常常同時用於兩個情境(很容易混淆)

  • 被組織如同塔狀的結構
  • (此處用法)沒有組織，但看起來像塔狀的結構
  1. 使用範圍: 記憶體編號高到低
  2. 無組織
  3. Heap Memory 被稱為 Resources(資源)
    需要時可動態拿取；不需要時可以自行釋放
  4. 程式無法直接存取 Heap 記憶體資源
    • 需要透過 Pointer 指標存取
  5. 需要使用 C++(new)、C(malloc) 來創建空間
  6. 不需要使用時，需要透過 C++(delete [])、C(free()) 來釋放空間
    • 可能造成位址丟失，該記憶體資源將永久卡住無法使用(Memory Leak)
    • 如果無釋放，當資源塞滿 Heap 會導致程式發生不可預期的情況

Code Section

  • 根據讀取的程式大小來動態調整
  • 放置函數的程式碼(e.g main function)

操作記憶體模式有

  • 靜態分配記憶體(Static Allocation): 只有 Stack
  • 動態分配記憶體(Dynamic Allocation): 可使用 Stack、Heap

3. 實體與邏輯資料結構

實體資料結構(Physical data structure) ，決定實際資料儲存在哪一塊記憶體，主要儲存方式為

Array

  • 固定大小(Size)
  • 可以被創建在 Stack 與 Heap (使用 pointer)

Linked List

  • 動態資料結構
  • 可動態調整大小
  • 只能被創建在 Heap (使用 pointer)
    • 在 Stack 中創建 Head 的指標指向第一個節點

邏輯資料結構(Logical data structure) ，決定依照什麼原則存取資料。(底層依然是由 array, Linked List 來實作)。主要方式為

線性資料結構(Linear)

  • Stack: 原則 LIFO
  • Queue: 原則 FIFO

非線性資料結構(non-Linear)

  • Tree: 原則 階層式
  • Graph: 原則 節點、連結式

表格式資料結構(Tabular)

  • Hash Table: 原則 Hash function

4. ADT 抽象資料結構

將較為底層的操作(如:位元等級的運算)隱藏，讓使用者只需要使用規定好的介面對資料進行存取異動即可。一般根據資料儲存資訊，資料操作來理解。

資料型別(Data type)

顯示資料(Representation of Data)

  • 資料如何被儲存
  • 舉例: int x；(被儲存 4 bytes = 32 bits)

資料操作(Operation on Data)

  • 可以對該資料進行何種操作
  • 舉例: -, +, *, /, %, ++, --

舉例: ADT(抽象型別)

顯示資料

  • 儲存資料的空間 int *A (儲存在 Heap 動態分配)
  • 最大容量(Capacity) int maxSize
  • 目前存放多少資料(size) int length

資料操作

  • 新增
    • add(element) / append(element)
    • add(index, element) / insert(index, element)
  • 刪除
    • remove(index)
  • 設置
    • set(index, element) / replace(index, element)
  • 搜尋
    • get(index)
    • search(key) / contains(key)
  • 排序
    • sort()

Doxygen 文檔建立系統

Doxygen介紹

Doxygen 是一種開源跨平台的文檔建立系統，(類似 javaDoc)。完全支持 C/C++、Java、Object-C

優點

  • 可從一組帶有註解的程式中生成 HTML 格式或是 LATEX 格式的文檔，亦可產生 pdf Unix man 等手冊文件。因為是從程式碼註解中直接提取文件，容易保持格式的一致性。
  • 可提取程式碼的結構，通過包含依賴圖(Dependency Graphs)、繼承圖(inheritance Diagram)、協作圖(Collaboration Diagram)等自動生成的圖，可使不同程式模組關係視覺化。
  • 可輸入格式: html, xml, LaTeX, PDF, RTF, Unix man

官網: Doxygen

基礎操作

安裝 Doxygen (Ubuntu)

sudo apt-get install doxygen

指令

Doxygen 可分為兩部分

  1. 特定格式註解的寫法(主要)
  2. 利用Doxygen工具來產生文件

產生一份 Doxygen 預設的 Config 檔案，並以此檔案來進行修改

  • doxygen -g [config-file]: 會產生Doxyfile(一個檔名叫做 “[config-file]”的預設 config 檔)
  • doxygen -u [config-file]: 更新舊的配置文件
  • doxygen [config-file]: 根據Doxyfile樣式，生成文檔

生成不同風格的文件，如 rtf, html, LaTeX

  • RTF格式: doxygen -w rtf styleSheetFile
  • HTML格式: doxygen -w html headerFile footerFile styleSheetFile [config-file]
  • LaTeX格式: doxygen -w latex headerFile StyleSheetFile [config-file]

註解格式

一般來說，要為每個 class 和 class 中重要的成員函數增加短註解和長註解。

  • 短註解給出類別或函數的基本訊息的描述
  • 長註解則是更加完整的描述。
  1. 類別的短註解和長註解與成員函數的簡短描述，放在標頭檔中。
  2. 成員函數的長註解將出現在成員函數的實現出現的位置

Scoop 套件管理軟體

Scoop Windows套件管理軟體

一個 windows 的軟體管理工具，用於安裝各式各樣的環境，類似於 Linux 的 apt-get。
相似的軟體有 :

  • Chocolatey
  • Ninite
  • CoApp。

Scoop 專注於在 Command-line 的使用，可以安裝軟體不需要一直挑選網頁下載。

快速安裝

要求 Requirement

  • PowerShell 3
  • 允許 PowerShell 可執行本地腳本

Installing Scoop

確認上方無誤後，在 powershell 中打入安裝指令。

如果想要安裝到指定的目錄 (如 : D: \Applications\Scoop)

Scoop 的使用

主要常用的指令有下列幾個 :

  • scoop help
    • 查看有哪些指令可用
  • scoop install {app name}
    • 安裝 app
  • scoop search {app name}
    • 搜尋確認是否有此 app ，或觀察目前版本
  • scoop update
    • 更新 scoop ，後方可加上 app 名稱，表示更新 app。 如果使用 * 則是安裝的全部都更新
  • scoop cleanup {app name}
    • 刪掉目前沒用或較舊的 app 版本
  • scoop uninstall {app name}
    • 刪掉現有的 app
  • scoop uninstall scoop
    • 刪掉他自己

Concepts

Apps

Scoop 使用名稱 “App” 表示一個應用程式或是需要被安裝的程式。一個 “App”應該是一個獨立的單元，也就是說一個”App”可能包含一個或是多個可執行的腳本。

Buckets

一個 Bucket 像是一個容器，裡面有包含很多個可安裝的 “App”。準確來說是一個 github 的 repository，其中包含用來描述安裝 App 的 JSON 語法，

語法

  • 調查已知的 bucket，可以直接觀察檔案 “buckets.json”，或是打入指令

  • 安裝其他的 bucket 到電腦中

  • 創建自己的 bucket
    使用 github 的 repository，

  2. 加入 app 到你的 bucket 中
  3. 設定你的 scoop 去連結你自己新建立的 bucket :
  4. 檢查是否成功`

2. 命令

2.2 DDL-數據定義語言

  • CREATE: 創建一個新的表。一個表的視圖，或是資料庫中其它的對象

  • DROP: 刪除整的表單，或是表的視圖。或資料庫中其它的對象。

2.3 DML-數據操作語言

  • INSERT: 創建一條紀錄
  • UPDATE: 修改紀錄
  • DELETE : 刪除紀錄

  • 2.4 DQL-數據查詢語言

  • SELECT: 從一個或多個表中檢索某些紀錄

SQLite(1) 介紹

SQL與MySQL差異

兩者大致功能差異

  • SQLite: 講究功能簡約、小型化、追求最大磁碟效率
  • MySQL: 功能全面、綜合化、追求最大並行處理效率

使用時機

  • SQLite: 單機上使用，需要方便多方移植，頻繁讀寫
  • MySQL: 滿足多用戶同時訪問，訪問量較大

SQLite 介紹

SQLite是一個基於文件的資料庫，提供了處理多種數據類型的工具集，使用上比起伺服器運作的Process型伺服器較為輕鬆。

當一個應用程式使用SQLite時，應用程式會直接訪問文件所在位址(SQLite資料庫)，而非通過連接埠(port, socket)來交互呼叫訪問。使得SQLite較為快速與高效。SQLite是一個實現自給自足的函數庫，沒有伺服器，不須配置的SQL資料庫引擎。

SQLite支持的數據類型

  • NULL: null值
  • INTEGER: 有符號整數，按照設定用1、2、3、4、6、8字節儲存
  • READ: 浮點數，使用8字節IEEE浮點數方式儲存
  • TEXT: 字串，使用資料庫編碼(UTF-8，UTF-16BE，UTF-16LE)儲存
  • BLOB: 二進制檔案，如何輸入就如何輸出

SQLite 優點與缺點

優點

  • 基於文件: 整個資料庫都在磁碟中的一個文件中，有很好的遷移性
  • 標準化: 支持SQL。但略去一些功能。
  • 易於開發、測試: 有靈活性，且簡潔到只需一個文件和C連結庫即可。

缺點

  • 無用戶管理: 高級資料庫能支持用戶系統，如:能管理資料庫和table的訪問權限。SQLite無此功能。
  • 缺乏額外優化性能的活躍性: 因為不支持各種技巧來提升效能。所以容易配置與使用，但因為不複雜，所以無法更快。

SQLite使用時機

  1. 嵌入式應用: 強調遷移性，不需多餘的擴展的應用，如: 單用戶的本地應用，移動應用，遊戲
  2. 代替磁碟訪問: 需要頻繁直接讀寫硬碟文件的應用。
  3. 測試: 易於建置與攜帶

SQLite不適合應用

  1. 多用戶應用: 如果應用程式需要被多用戶同時訪問與存取，則功能上無法完整支援
  2. 需要大面積寫入數據的應用: SQLite缺陷是寫入操作，同一時間只允許一個寫操作，因此throughput有限。
